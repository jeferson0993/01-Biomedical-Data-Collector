# Plataforma Integrada de Coleta de Dados Biológicos: Uma Arquitetura para Curadoria Automatizada de Repositórios Públicos

## Introdução

Na contemporaneidade, o volume de dados biológicos depositados em repositórios públicos cresce em ritmo exponencial. O Gene Expression Omnibus (GEO), o NCBI Gene, o PubMed e o UniProt acumulam diariamente milhares de novos registros — sequências, perfis de expressão, artigos e anotações funcionais. Em primeiro lugar, esse cenário representa uma oportunidade ímpar para a pesquisa biomédica. Em contrapartida, a diversidade de formatos, protocolos de acesso e modelos de metadados impõe uma barreira significativa: cada fonte exige um coletor específico, tratamento distinto de rate limiting e lógica própria de validação. Sobretudo, a ausência de padronização no armazenamento compromete a reprodutibilidade dos estudos. Muito se discute sobre a necessidade de ferramentas integradas que unifiquem a extração, a curadoria e a persistência desses dados. É fato que soluções ad hoc, como scripts isolados para cada repositório, geram retrabalho e dificultam a manutenção. Antes de mais nada, é preciso estabelecer uma arquitetura que trate a coleta como um serviço — não como um conjunto de artefatos dispersos.

A princípio, este trabalho apresenta o Projeto 1 da Plataforma Integrada de Bioinformática: um Biomedical Data Collector acoplado a um Data Lake, ambos conteinerizados. O objetivo é oferecer uma API REST que dispare coletas assíncronas nas quatro fontes mencionadas, armazene os dados brutos de forma imutável no MinIO e persista os metadados no PostgreSQL, tudo com rastreabilidade e resiliência.

## Desenvolvimento

A arquitetura do sistema divide-se em três camadas principais. A primeira é a camada de **API e orquestração**, implementada com FastAPI. Seus cinco endpoints — `POST /collections`, `GET /collections`, `GET /collections/{id}`, `GET /collections/{id}/download/{dataset_id}` e `GET /health` — expõem uma interface limpa para disparar e monitorar coletas. A segunda camada é o **motor de coletores**, composto por classes abstratas e implementações concretas para GEO, NCBI Gene, PubMed e UniProt. Cada coletor segue o mesmo ciclo: fetch, validate, upload e metadata. A terceira camada é o **Data Lake**, sustentado por MinIO (armazenamento de objetos) e PostgreSQL (metadados relacionais). O Docker Compose orquestra os três serviços — api, postgres e minio — mais um job one-shot que cria o bucket raw.

Além disso, o fluxo de uma coleta foi projetado para ser resiliente. Uma vez que o usuário envia um POST com `source` e `external_id`, o sistema cria um registro `Collection(status=pending)` no PostgreSQL e dispara uma tarefa assíncrona em segundo plano. Do mesmo modo, essa tarefa atualiza o status para `running`, realiza o download da fonte externa com retry exponencial e backoff configurável, valida o formato e o conteúdo, faz o upload para o MinIO no caminho `raw/{source}/{external_id}/` e gera o arquivo `metadata.json` lado a lado com os dados. Por fim, insere os registros `Dataset` correspondentes e marca a coleta como `completed`. Em caso de falha, o status é alterado para `failed` e a mensagem de erro é preservada.

Por outro lado, a escolha tecnológica não foi trivial. O gerenciador de dependências `uv` foi selecionado por sua velocidade e lockfile reproduzível. O SQLAlchemy 2.0 no modo assíncrono, combinado com Alembic para migrations, fornece uma camada de dados robusta. O Pydantic v2, por ser nativo do FastAPI, reutiliza os mesmos schemas para validação e documentação automática. Os testes com `pytest`, `httpx`, `pytest-asyncio` e `respx` permitem mockar chamadas HTTP externas sem depender dos repositórios reais. O `ruff` unifica lint e formatação, enquanto o `mypy` garante a segurança de tipos.

Como resultado, a estrutura de diretórios reflete essa separação de responsabilidades. O diretório `app/` contém os submódulos `models/`, `schemas/`, `api/`, `collectors/`, `storage/` e `utils/`. Cada coletor reside em seu próprio pacote — `geo/`, `ncbi_gene/`, `pubmed/`, `uniprot/` — facilitando a adição de novas fontes no futuro. Os testes seguem a mesma organização, com módulos específicos para coletores, API e armazenamento.

Não obstante, algumas decisões merecem destaque pela sua relevância para a reprodutibilidade científica. Em primeiro plano, a imutabilidade dos dados brutos: uma vez armazenados no MinIO, eles não podem ser alterados. Cada coleta gera novos arquivos, mesmo que o `external_id` já tenha sido coletado anteriormente. Do mesmo modo, o arquivo `metadata.json` acompanha cada conjunto de dados com os campos `source`, `date`, `version` e `parameters`, garantindo a procedência da informação. A fixação da semente aleatória — `random.seed(42)` — assegura que eventuais amostragens sejam reproduzíveis. Por conseguinte, pesquisadores podem confiar que os dados de hoje serão idênticos aos de amanhã para o mesmo conjunto de parâmetros.

O modelo de dados, por sua vez, é enxuto e funcional. A tabela `Collection` armazena o identificador UUID, a fonte (`geo`, `ncbi_gene`, `pubmed`, `uniprot`), o ID externo, o status (`pending`, `running`, `completed`, `failed`), o caminho no MinIO, os metadados em JSONB, a mensagem de erro e os timestamps de criação e atualização. A tabela `Dataset`, vinculada por chave estrangeira à `Collection`, registra cada arquivo individual — nome, formato, tamanho, checksum SHA-256 e caminho completo no MinIO.

Em vista disso, o pesquisador pode consultar o progresso de qualquer coleta com chamadas HTTP simples. Um script em Bash com `curl` e `jq` pode disparar a coleta, fazer polling até a conclusão e listar os datasets gerados — tudo em poucas linhas. Clientes Python, tanto síncronos quanto assíncronos, também são suportados, conforme demonstrado na documentação de exemplos de uso.

## Conclusão

Portanto, o Projeto 1 estabelece a fundação sobre a qual os demais 15 projetos da plataforma serão construídos. Em suma, ele resolve o problema da dispersão de fontes de dados biológicos ao oferecer uma API unificada, armazenamento imutável e metadados padronizados. Nesse sentido, os projetos de genômica, transcriptômica, proteômica, descoberta de biomarcadores, redes biológicas, docking molecular e todos os demais poderão consumir dados prontos e rastreáveis diretamente do Data Lake. Dessa forma, eliminam-se gargalos operacionais e garante-se que a base de qualquer análise — os dados brutos — seja íntegra, documentada e reprodutível. Em resumo, mais do que um coletor de dados, o Projeto 1 é a espinha dorsal de toda a Plataforma Integrada de Bioinformática.
